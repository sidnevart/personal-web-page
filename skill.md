# SKILL — Build a world-class portfolio website for a senior-caliber backend engineer / startup builder

## Роль

Ты — elite product designer + staff frontend engineer + UX strategist + conversion copywriter + art director.
Твоя задача — не просто сверстать сайт, а создать **портфолио как продукт**, который:

1. повышает мою ценность на рынке,
2. продаёт меня как сильного разработчика,
3. помогает получать офферы, клиентов, партнёров, OSS-возможности и входящие сообщения,
4. показывает не “набор технологий”, а **реальный инженерный impact**.

Ты работаешь как человек, который делает сайт уровня top-tier digital studio, но при этом думает как основатель и hiring manager.

---

## Кто я и как меня надо позиционировать

Позиционируй меня не как “просто backend-разработчика”, а как сочетание ролей:

* **Big Tech Engineer**
* **Open Source Builder**
* **Startup Engineer / Technical Founder Type**
* **Product-minded Backend / Systems Engineer**
* **Engineer who can build revenue-impact systems for business**

Сайт должен транслировать такой образ:

* я умею делать сложные инженерные системы;
* я не только пишу код, но и думаю о продукте, бизнесе, UX и результате;
* я умею запускать свои проекты;
* я умею делать решения для бизнеса;
* я способен быть сильным индивидуальным исполнителем и техническим партнёром;
* у меня есть вкус, инженерная глубина и способность доводить до результата.

Не делай образ “очередной frontend-fashion portfolio”.
Нужен образ: **умный, сильный, технологичный, дорогой, современный, надёжный**.

---

## Главная цель сайта

Сайт должен отвечать на вопрос посетителя за 5–10 секунд:

**“Почему именно с этим человеком стоит работать?”**

И сразу доносить:

* чем я занимаюсь,
* что я умею,
* какие проекты умею делать,
* какой у меня опыт,
* почему мне можно доверять,
* как со мной связаться.

---

## Основной принцип дизайна

Нужен стиль уровня:

* premium tech,
* minimal, but not empty,
* editorial + product,
* cinematic, but usable,
* modern, but not trend-trash,
* tasteful motion,
* strong typography,
* excellent spacing,
* clean visual hierarchy,
* expensive look without visual noise.

Избегай:

* дешёвых градиентов ради градиентов,
* типовых “agency templates”,
* перегруженного glassmorphism,
* кринжовых 3D-иллюстраций,
* generic AI-looking art,
* бессмысленных анимаций,
* тяжёлых эффектов, которые убивают UX,
* ощущения “красиво, но непонятно чем человек вообще занимается”.

## Что нужно реализовать

### 1. Главная страница

Главная страница должна быть очень сильной и продавать меня с первого экрана.

Обязательные блоки:

1. **Hero section**

   * сильный headline;
   * понятный subheadline;
   * короткое позиционирование;
   * CTA: посмотреть проекты / связаться / обсудить проект;
   * быстрые proof-points: big tech, startup building, open source, business projects.

2. **Selected work / featured projects**

   * 3–6 лучших проектов;
   * красивый preview;
   * кратко: проблема → решение → результат → стек → моя роль;
   * переход на отдельную project page.

3. **About / positioning**

   * кто я;
   * как я думаю;
   * что я умею решать;
   * почему я полезен бизнесу и продукту;
   * чем отличаюсь от обычного “coder for hire”.

4. **What I build**

   * категории решений:

     * internal tools,
     * backend platforms,
     * AI products,
     * startup MVP,
     * analytics / infra / developer tooling,
     * business software.

5. **Experience**

   * big tech;
   * open source;
   * personal products / startups;
   * client/business projects.

6. **Contact / CTA**

   * форма;
   * email;
   * Telegram / LinkedIn / GitHub;
   * понятный сценарий связи.

---

### 2. Страница проектов

Сделай отдельный полноценный раздел **Projects**.

Там должны быть:

* фильтры/категории;
* качественные карточки;
* понятное описание;
* статус проекта:

  * shipped,
  * in progress,
  * open source,
  * concept / experimental;
* технология, домен, роль, impact.

Карточки должны выглядеть как работы сильного продуктового инженера, а не как “список репозиториев”.

---

### 3. Отдельные страницы проектов

Для каждого важного проекта должна быть отдельная page.

Шаблон project page:

1. Название проекта
2. Краткий one-line summary
3. Контекст / проблема
4. Что было сделано
5. Моя роль
6. Архитектурное или продуктовое решение
7. Challenges / constraints
8. Результат / эффект / impact
9. Tech stack
10. Галерея / визуалы / схемы, если уместно
11. Ссылки: demo / GitHub / case study / contact

Для big tech backend-проектов **не придумывать фейковые UI-скриншоты**.
Там лучше использовать:

* аккуратные архитектурные визуалы,
* стильные diagram-like blocks,
* terminal / infra / data-flow presentation,
* метрики, impact cards,
* backend-oriented visual storytelling.

Для pet/startup/client/OSS проектов допустимы:

* реалистичные device mockups,
* dashboard-style previews,
* product shots,
* UI scenes.

## Правила для изображений и визуалов

Если у проекта нет хороших исходных картинок, нужно создавать **реалистичные, премиальные, believable visuals**, которые выглядят как настоящие product renders.

Требования к визуалам:

* не должно быть видно, что картинка “навайбкожена”;
* визуал должен быть правдоподобным;
* стиль должен быть консистентным по всему сайту;
* без дешёвого AI-art вайба;
* без кривого текста в изображениях;
* без перегруженных сцен.

Подход к визуалам:

* для SaaS / startup / app проектов — реалистичные product mockups;
* для AI / analytics / dashboards — clean dashboard renders;
* для backend / infra / systems — abstract but plausible system visuals, topology blocks, terminal-grade aesthetics;
* для open source — code/product hybrid presentation;
* для big tech internal systems — без fake UI, только tasteful engineering storytelling.

Если картинка проекту не нужна — не добавляй её насильно.

## UX / UI требования

Сайт должен ощущаться как сайт человека, у которого:

* сильный вкус,
* сильная инженерная база,
* высокий стандарт качества,
* понимание продукта,
* внимание к деталям.

Требования:

* идеальная визуальная иерархия;
* аккуратная сетка;
* крупная сильная типографика;
* много воздуха;
* очень хороший mobile UX;
* понятная навигация;
* быстрый time-to-value;
* сильные CTA;
* хорошая читаемость;
* продуманная работа с hover/focus/active states;
* анимации дорогие и спокойные;
* transitions не мешают сканированию контента;
* сайт должен быть “wow”, но без потери конверсии.

## Темы и визуальные режимы

Нужно реализовать:

* **dark mode**
* **light mode**
* автоматическое определение системной темы
* ручной переключатель
* корректное сохранение выбора пользователя
* отсутствие визуального дёргания при первом рендере
* обе темы должны быть реально красивыми, а не “одна норм, другая для галочки”

Используй системные возможности `prefers-color-scheme` и `color-scheme`, а палитры строй так, чтобы они оставались читаемыми и контрастными. ([MDN Web Docs][2])

## Локализация

Нужно поддержать 3 языка:

* **RU**
* **EN**
* **ES**

Важно:

* это не тупой буквальный перевод;
* тексты должны быть **transcreated** под язык и аудиторию;
* английская версия должна звучать как сайт сильного international engineer;
* испанская версия должна быть естественной и аккуратной;
* локаль должна влиять на:

  * тексты,
  * навигацию,
  * метаданные,
  * SEO,
  * URL / routing при необходимости.

Используй современную i18n-архитектуру для Next.js App Router и `next-intl`, с нормальным locale-based routing и locale-aware navigation. ([Next.js][3])

## SEO / discoverability / AI-readability

Сайт должен быть сделан так, чтобы его хорошо понимали:

* люди,
* поисковики,
* AI systems / crawlers.

Обязательно:

* качественные title/description/open graph;
* semantic HTML;
* clean heading hierarchy;
* project pages indexable;
* structured data / JSON-LD;
* person/entity description;
* article/case-study style metadata там, где уместно.

Используй JSON-LD и structured data для описания меня как профессионала, сайта как сущности и project/case-study страниц как контента, понятного поиску и AI. ([Next.js][4])

## Performance требования

Сайт должен быть **очень быстрым**.

Нельзя делать “дизайн ради дизайна”, который убивает производительность.

Цели:

* excellent perceived performance,
* fast initial load,
* minimal layout shift,
* smooth interaction,
* optimized images,
* lazy loading only where it makes sense,
* minimal JS on critical path,
* careful animation budget,
* no bloated dependencies.

Ориентируйся на актуальные Core Web Vitals: **LCP, INP, CLS** и строй решение так, чтобы сайт проходил их на хорошем уровне. ([web.dev][5])

---

## Accessibility требования

Сайт должен быть не “формально доступным”, а реально качественным:

* keyboard accessible;
* focus visible;
* good contrast;
* readable typography;
* sane motion;
* semantic landmarks;
* proper labels;
* screen-reader friendly structure;
* language attributes;
* no inaccessible gimmicks.

Цель — **WCAG 2.2 AA**. Также учитывай системные пользовательские предпочтения, включая контрастность и цветовую схему. ([W3C][6])

* уверенный;

* без кринжа;

## Формат работы

Работай по шагам:

### Шаг 1. Product framing

Сначала сформулируй:

* моё позиционирование,
* целевую аудиторию сайта,
* goals of the website,
* content strategy.

### Шаг 2. Information architecture

Предложи структуру страниц и user flow.

### Шаг 3. Visual direction

Опиши 2–3 сильных art directions и выбери лучший.

### Шаг 4. Design system

Определи:

* typography,
* spacing,
* color tokens,
* themes,
* card patterns,
* buttons,
* nav,
* motion principles.

### Шаг 5. Content mapping

Преобразуй мой опыт и проекты в контент.

### Шаг 6. Build

Собери сайт production-quality.

### Шаг 7. Polish

Дополируй до уровня “could actually ship”.
